# 프롬프트 엔지니어링 — 분류기 프롬프트 설계 의사결정

본 문서는 `src/domain/classification/classificationPrompt.ts` 에 적용된 프롬프트 설계의 근거와 검토된 대안을 정리한다. 측정 결과는 `classifier-evaluation.md` §5 참조.

---

## 1. baseline 프롬프트 (`implementation-plan.md` §4.5)

```text
당신은 사용자의 생산성 도구 사용 맥락을 분석하는 분류기입니다.

사용자가 작성한 메모 또는 일정의 제목과 내용을 보고 다음 user_type 중 하나를 선택하세요.

[user_type]
- 대학생: 강의, 시험, 과제, 팀플, 동아리, 학점, 수업 등 학업 맥락
- 직장인: 회의, 보고, 미팅, 업무, 결재, 출장 등 조직 내 업무 맥락
- 프리랜서: 클라이언트, 프로젝트 납품, 견적, 외주, 개인 사업 맥락
- 팀 사용자: 우리 팀, 부서, 공동, 협업, 스프린트 등 복수 인원 협업 맥락
- unknown: 사적 메모이거나 맥락이 모호한 경우

[입력]
제목: ${input.title}
내용: ${input.content}
타입: ${input.type}

[출력]
JSON으로만 답변하세요.
{
  "user_type": "대학생|직장인|프리랜서|팀 사용자|unknown",
  "confidence": 0.0,
  "reasoning": "판단 근거 한 문장",
  "keywords": ["감지 키워드"]
}
```

## 2. baseline 진단 — 왜 unknown 이 깨지는가

baseline 측정 (`gemma4:e4b`, QA §C-3 75%) 의 오답 3건 패턴:

| 입력 | 기대 | 실제 | confidence |
|---|---|---|---|
| 회의 준비 / (빈 내용) | unknown | 직장인 | 0.90 |
| 프로젝트 기획 / 시장조사 먼저 | unknown | 직장인 | 0.80 |
| 책 독서 메모 / 행동 경제학 3장 | unknown | 대학생 | 0.90 |

추출되는 구조적 문제 5가지:

| # | 문제 | 설명 |
|---|---|---|
| 1 | **unknown 정의가 약함** | "사적 메모이거나 맥락이 모호한 경우" 한 줄. '모호함' 의 기준이 없어 모델이 임의 해석 |
| 2 | **분류 강제 압력** | "다음 user_type 중 하나를 선택하세요" 는 unknown 도 동등 선택지라는 신호가 약함. 모델이 "추측이라도" 4 클래스 중 하나에 배정 |
| 3 | **키워드 단편 매칭 유도** | 클래스 정의가 키워드 나열로만 구성되어 '회의' 1개, '경제학' 1개로 점프하는 first-match 행동을 유도 |
| 4 | **confidence 캘리브레이션 가이드 없음** | 0.0~1.0 범위만 주고 어떤 상황에서 어떤 값을 줄지 명시 없음. 결과적으로 0.80~0.95 로 자기과신 → 0.75 threshold 통과 → 파이프라인 fallback 도 무력화 |
| 5 | **빈 content / 짧은 content 처리 가이드 부재** | qa-05 같은 빈 내용 케이스에서 모델이 제목 단어 하나로 결정 |

## 3. 검토된 후보 5종

### 3.1 후보 1 — unknown 정의 강화 + 분류 강제 압력 제거

**근거**: 진단 1·2·5. unknown 이 동등 선택지임을 명시하면 모델이 추측 대신 unknown 으로 빠짐.

**변경**:
- 도입부 "다음 user_type 중 하나를 선택하세요" → "가장 적합한 하나를 선택하되, 단서가 약하면 반드시 unknown 을 선택합니다"
- unknown 정의를 다음 트리거 4개 명시로 교체:
  1. 사적 영역 (독서·운동·여가·소비·여행·취미·가계) 단서만 있을 때
  2. 내용이 비었거나 단일 행위 단어("준비", "정리", "기획") 만 있을 때
  3. 두 개 이상 클래스의 단서가 비슷한 강도로 충돌할 때
  4. 학업·업무·외주·팀 협업 어느 쪽으로도 단정할 단서가 없을 때

**겨냥 케이스**: qa-05, qa-06, qa-12 — 진단의 unknown 오답 3건 전부.

**리스크**: unknown 과다 → 명확한 4 클래스에서 false negative 발생 가능. edge 케이스 (학회, 사내 워크샵) 도 unknown 으로 도망갈 수 있음.

**완화**: 트리거 조건을 명시적으로 enumerate (위 4개) 해서 일반어 keyword 매칭에 의존하지 않게.

### 3.2 후보 2 — Few-shot 예시 (특히 unknown 음성 사례)

**근거**: 진단 3. 자연어 정의보다 예시가 모델의 실제 행동에 더 강하게 작용. 특히 "이 키워드가 보여도 unknown 인 경우" 의 음성 사례를 제시하면 first-match 행동이 차단됨.

**변경**: 프롬프트 끝에 4~6 예시 삽입. 각 클래스 1개 + **unknown 음성 사례 2개**.

```text
[예시]
- "강의실에서 자료구조 챕터 정리" → 대학생 (확신 0.9, 근거: 강의/챕터)
- "회의 준비" (빈 내용) → unknown (확신 0.4,
  근거: '회의' 단어만으로는 업무·동아리·동호회 모두 가능)
- "행동 경제학 3장 메모" → unknown (확신 0.4,
  근거: '경제학' 은 책 주제이지 학업 단서 아님)
- ...
```

**겨냥 케이스**: qa-05 ('회의' 단어 트랩), qa-12 ('경제학' 단어 트랩).

**리스크**:
- 평가셋과 표현이 겹치면 데이터 누수에 가까운 과적합 → 평가셋 외 신규 입력에서 성능 저하
- 프롬프트 토큰 증가로 작은 모델에서 응답 안정성 저하 가능

**완화**: 예시는 평가셋과 어휘가 겹치지 않게 다른 단어 선택. 음성 사례는 2개로 제한.

### 3.3 후보 3 — confidence 캘리브레이션 가이드

**근거**: 진단 4. baseline 에서 오답 confidence 평균 0.867 — 0.75 threshold 모두 통과해 `resolveUserType` 의 history fallback 경로가 활성화되지 않음. 점수가 오답에서 떨어지면 raw 오류가 파이프라인 정답으로 회복됨.

**변경**: `confidence` 자리에 구간별 정의 명시.

```text
[confidence 산정]
- 0.9~1.0: 클래스 단서 키워드 2개 이상 + 다른 해석 어려움
- 0.7~0.9: 주된 단서 1개, 보조 단서 약함
- 0.5~0.7: 단서가 약하거나 1개 — 이 구간이면 unknown 후보로 재검토
- 0.5 미만: 단서 없음 — 반드시 unknown
```

**겨냥 케이스**: qa-05/06/12 — raw label 이 틀려도 confidence < 0.75 로 떨어지면 파이프라인이 fallback 으로 회복.

**리스크**: 캘리브레이션 가이드는 모델별 순응도 편차가 큼. 특히 작은 모델 (3B 이하) 은 무시하고 0.9 만 출력하는 경향.

**완화**: 단독 사용보다 후보 1·5 와 결합. raw 정확도와 캘리브레이션을 동시에 개선.

### 3.4 후보 4 — Decision rule 단계화

**근거**: 진단 3. 모델이 키워드 first-match 로 점프하는 행동을 막으려면 결정 순서를 강제해야 함. JSON-only 응답이라 외부에 노출 안 되지만, 내부 추론 순서를 가이드하면 reasoning 필드와 user_type 의 정합성이 높아짐.

**변경**: 클래스 정의 위에 결정 절차 명시.

```text
[판단 절차]
다음 순서로 판단합니다:
1. 사적 활동(취미·소비·여가·독서·운동) 인가? → 예 → unknown
2. 내용이 5자 미만이거나 비었는가? → 예 → 제목만으로 강한 단서가 없으면 unknown
3. 학업/조직 업무/외주/팀 협업 단서가 있는가? → 가장 강한 쪽 선택
4. 두 클래스 단서가 비슷한 강도면 → unknown
```

**겨냥 케이스**: qa-05 (절차 2 — 빈 내용), qa-06 (절차 3·4 — 일반어), qa-12 (절차 1).

**리스크**: 작은 모델은 절차를 무시하고 마지막 항목만 따르는 경우. 또 reasoning 필드가 절차를 그대로 복사해 단조로워질 수 있음.

**완화**: reasoning 을 "한 문장" 으로 강제 → 절차 복사 방지.

### 3.5 후보 5 — 함정 키워드 / Negative criteria

**근거**: 진단 3. baseline 오답이 모두 단일 단어 함정 ('회의', '경제학') 에서 발생. 클래스 정의에 "이 단어가 있어도 이 클래스가 아닐 수 있다" 를 명시하면 직접적 차단 효과.

**변경**: 각 클래스에 negative criteria 추가.

```text
[클래스별 함정 단어]
- 직장인: '회의·미팅·프로젝트' 단독으로는 분류 금지 (취미 모임/사적 일정 가능)
- 대학생: '공부·학습·책·독서·~학·~론' 단독으로는 분류 금지 (자기계발 가능)
- 팀 사용자: '회의·리뷰' 단독으로는 분류 금지 — '우리 팀/부서/공동' 단서가 함께 있어야 함
- 프리랜서: '프로젝트' 단독으로는 분류 금지 — '클라이언트/외주/납품/견적' 단서 필요
```

**겨냥 케이스**: qa-05 ('회의' 단독), qa-12 ('경제학' = ~학) — 직격.

**리스크**:
- 리스트가 길어지며 토큰 증가, 작은 모델에서 응답 안정성 저하
- 함정 키워드 enumerate 가 평가셋에 맞춰지면 다른 함정에 무력함

**완화**: 4 클래스 × 1줄 = 4줄로 제한. 평가셋 키워드만 나열하지 말고 같은 부류 (예: 책 주제어 = ~학/~론/~사) 로 일반화.

## 4. 권장 적용 순서 (검토 시점 기준)

| 단계 | 후보 | 예상 효과 | 누적 토큰 증가 |
|---|---|---|---|
| 1 | 후보 1 (unknown 강화) | qa-05/06/12 직접 회복, 80% 통과 가능 | +6줄 |
| 2 | 후보 5 (함정 키워드) | unknown 강화가 over-trigger 되는 경우 보강 | +4줄 |
| 3 | 후보 3 (캘리브레이션) | raw 가 틀려도 파이프라인 회복 — 안전망 | +4줄 |
| 4 | 후보 4 (decision rule) | 추가 안정화 — 위 3개로 부족할 때 | +5줄 |
| 5 | 후보 2 (few-shot) | 마지막 보루 — 평가셋 누수 위험 가장 큼 | +6~10줄 |

**1차 시도로 채택된 조합**: **1 + 5**.
- 진단의 가장 큰 문제(unknown 정의·함정 키워드)를 직접 겨냥
- 추가 토큰 +10줄로 작은 모델 영향 작음
- few-shot 누수 회피
- 부족하면 단계적으로 3 → 4 추가, 최후에 2

| + `think:false` 결합 | **100%** | 100% (33건) → **98.7% (152건)** | 1.000 → 0.986 |

상세 측정 데이터는 `classifier-evaluation.md` §5.3 (후보 1+5 적용) / §5.4 (think:false 결합) 참조.

요약:
- QA §C-3 정확도: 75% → 91.7% → **100%**
- unknown 회복: 1/4 → 4/4 (100%)
- 함정 키워드 정확히 작동 (qa-05/06/12 회복)
- 152건 확장에서도 98.7% 유지 (오답 2건은 약점 카테고리 noise)

## 7. 모델 호환성 — 프롬프트가 모델별로 다르게 작동

별도 측정 (`scaling-and-cost-analysis.md` §3 참조) 에서 같은 프롬프트로 다른 모델 swap 시도 결과:

| 모델 | QA §C-3 정확도 | 패턴 |
|---|---|---|
| `qwen2.5:3b` (1.9GB) | **41.7% FAIL** | 함정 키워드 over-trigger — 모든 명확 케이스를 unknown 으로 |
| `qwen2.5:7b` (4.7GB) | 75% FAIL | 함정 키워드 under-trigger — 진단의 baseline 패턴 재현 |
| `gemma2:9b` (5.4GB) | 91.7% PASS | 한 케이스 (qa-09) 만 unknown 으로 보수적 회귀 |
| **`gemma4:e4b` (9.6GB)** | **100% PASS** | 현재 default |

**핵심 시사점**: 본 프롬프트는 **gemma 계열에 최적화됨**. 모델 swap 시 정확도 100% 가 깨질 수 있음.

이유 추정:
- 함정 키워드 negative criteria 는 "동시에 여러 단서를 종합 평가" 능력을 요구
- 큰 모델 (8B+) 은 종합 판단, 작은 모델 (3B) 은 단순 매칭 → over-trigger
- gemma 시리즈는 instruction following 이 강해 negative criteria 충실히 따름
- qwen 시리즈는 결정 빠르고 자기 추론 강해 가이드 무시 가능성

## 8. 향후 보강 후보 (선택)

### 8.1 `worker-code-01` / `worker-slang-01` 재발 방지

평가셋 152건의 오답 2건은 다음 카테고리:

| id | 입력 | 기대 | 실제 |
|---|---|---|---|
| `worker-code-01` | "client meeting 준비" / "자료 review 부탁드림" | 직장인 | 프리랜서 |
| `worker-slang-01` | "재택 ㄱㄱ" / "오늘 자료 정리만 하면 됨" | 직장인 | unknown |

**보강 방안**:
1. 직장인 정의에 "재택근무·원격" 어휘 추가
2. 프리랜서 함정 키워드 강화: "한국어 '클라이언트' 가 있어야 프리랜서. 영어 'client' 만 있으면 직장인 가능성 높음"

다만 라벨 자체 모호성 있어 효과 한정적. 프롬프트 +3~5줄 vs 정확도 +1.3%p 의 trade-off.

### 8.2 모델 swap 대비 — 후보 3 (캘리브레이션) 도입

현재 gemma4:e4b 의존성이 강함. 향후 모델 변경 시 후보 3 (confidence 캘리브레이션) 추가하면 raw 오류가 fallback 으로 회복되는 경로가 살아남.

### 8.3 평가셋 확장 시 케이스 누수 방지

후보 2 (few-shot) 적용 시 예시 어휘가 평가셋과 겹치면 과적합. 평가셋 외 새 데이터 (베타 사용자 메모) 수집 후 generalization 검증 후에 후보 2 검토.

## 9. 결정의 핵심 — 프롬프트 vs 모델 vs 셋업

이번 측정 사이클에서 정확도 영향 요인 순위:

| 영향 | 변화 폭 | 비고 |
|---|---|---|
| **1. 프롬프트 (후보 1+5)** | +16.7%p | 75% → 91.7% |
| **2. 추론 모드 (`think:false`)** | +8.3%p | 91.7% → 100% (over-reasoning 차단) |
| 3. 모델 선택 | ±25%p | qwen2.5:3b 41.7% / gemma2:9b 91.7% / gemma4:e4b 100% |
| 4. 출력 schema 축소 | 0%p | latency 만 영향 |
| 5. 평가셋 확장 (33→152) | -1.3%p | 분포 변화로 어려운 케이스 노출 |

→ **모델 선택 + 프롬프트 + 추론 모드 3개가 본질**. schema 와 평가셋은 부수적.

→ **운영 진입 후 변경 위험도**: 모델 swap > 프롬프트 변경 > 추론 모드 변경 > schema. 모델 변경 시 반드시 152건 재측정 + 신뢰구간 확인.
