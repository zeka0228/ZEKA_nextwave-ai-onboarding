# NextWave AI 온보딩 가이드 시스템 기능 명세서 (v0.4 수정본)

> [!info] 문서 목적 및 범위
> 본 문서는 NextWave 서비스의 기능 사용률 저하 문제를 해결하기 위한  
> AI 기반 온보딩 가이드 시스템의 기능 명세를 정의한다.
>
> v0.4에서는 문제 정의, 데이터 근거, AI 활용 타당성, 추천 로직을 통합하여  
> 실제 서비스 적용 가능성과 평가 설득력을 강화하였다.
>
> v0.4-revised에서는 KPI 산정 근거를 본문에 통합하고, 섹션 번호 갱신에 따른 본문 내 교차 참조를 일괄 정정하였다.

## 0. 문제 정의 및 데이터 근거

### 0.1 문제 정의

NextWave는 일정 관리, 협업 메모, 팀 협업, 알림 자동화 기능을 제공하는 생산성 SaaS 서비스이다.

그러나 현재 서비스에서는 다음과 같은 문제가 발생하고 있다.

> 사용자가 서비스 내에서 다음에 어떤 기능을 사용해야 하는지 판단하지 못해  
> 핵심 기능(협업, 자동화)을 충분히 사용하지 못한다.

특히 신규 사용자 및 무료체험 사용자는

- 기능이 많지만 어디서 시작해야 할지 모르고
- 협업 기능을 언제 사용해야 하는지 이해하지 못하며
- 결과적으로 핵심 기능 사용으로 이어지지 않는다

---

### 0.2 데이터 기반 근거

#### 기능 사용 통계

| 기능 | 사용자 수 | 평균 사용 횟수 |
|---|---:|---:|
| 일정 관리 | 2,840 | 8.2 |

→ 해석

- 일정/메모는 진입 기능으로 잘 사용됨
- 팀 초대/알림은 핵심 기능임에도 사용률 낮음

| 이슈 | 비율 |

### 0.3 해결 전략

본 시스템은 다음 전략을 따른다.

1. 사용자의 첫 행동(메모/일정 작성)을 기준으로
2. 자연어 입력을 분석하여 사용자 맥락(user_type)을 추정하고
3. 사용자 행동 데이터(기능 사용 이력)를 결합하여
4. 아직 사용하지 않은 핵심 기능을 추천한다

즉,

> "사용자가 행동을 수행한 직후"  
> 다음 행동을 자연스럽게 연결하는 것이 핵심이다.

---

### 0.4 LLM 활용 이유

본 시스템은 rule-based 방식이 아닌 LLM 기반 분류를 사용한다.

그 이유는 다음과 같다.

- 메모/일정은 자유로운 자연어 입력이므로 규칙 기반 분류 한계 존재
- 동일 단어라도 맥락에 따라 의미가 달라짐
- 사용자 의도를 유연하게 해석하기 위해 LLM이 적합

따라서 본 시스템은

> "텍스트 맥락 분석 → user_type 추론" 단계에 LLM을 활용한다.

### 0.5 기대 효과 및 KPI

#### 기대 효과

| 문제 | 해결 방식 | 기대 효과 |
|---|---|---|
| 기능 탐색 어려움 | 생성 직후 가이드 제공 | 초기 UX 개선 |
| 팀 기능 사용률 낮음 | 협업 기능 추천 | 팀 초대 증가 |
| 무료체험 가치 부족 | 핵심 기능 경험 유도 | 전환율 증가 |
| 자동화 기능 미사용 | 알림 추천 | 리텐션 증가 |

#### KPI (목표)

| 지표 | 목표 |
|---|---|
| 팀 초대 사용률 | **+20%** |
| 알림 자동화 사용률 | **+15%** |
| 신규 사용자 핵심 기능 도달률 | **+25%** |

#### KPI 산정 근거

NextWave는 일반 사용자를 대상으로 하는 생산성 SaaS 서비스이므로, 본 프로젝트는 사용자층 특성에 부합하는 **B2C 온보딩 벤치마크**를 기준으로 KPI를 설정한다.

업계 자료에 따르면 B2C 서비스에서 양호한 온보딩 전환율은 **30~50% 수준**으로 보고된다.

> 출처: Userlist, "The One Metric to Rule Them All in User Onboarding"  
> https://userlist.com/blog/one-metric-user-onboarding-claudiu-murariu/

본 프로젝트의 초기 KPI는 위 벤치마크 범위의 **일부 이상을 달성하는 것을 목표**로 하되, AI 가이드 시스템의 첫 도입이라는 점을 고려하여 **30~50% 성공률 대비 보수적으로 설정**하였다.

즉, 위에 제시한 +20% / +15% / +25%는 운영 검증 이전까지의 **최소 달성 기준**이며, 실 운영 시 단계적으로 30~50% 벤치마크 범위에 도달하는 것을 상향 목표로 둔다. 이는 첫 도입 단계에서 무리한 수치를 약속하지 않으면서도, 업계 표준 범위 안에서 점진적으로 성과를 검증해 나가는 접근이다.

## 1. 용어 정의

| 용어 | 정의 |
|---|---|
| user_type | 사용자 분류 라벨 (대학생 / 직장인 / 프리랜서 / 팀 사용자 / 개인 사용자) |
| 가이드(Guide) | 사용자에게 다음 행동을 추천하는 UI 요소 |
| 트리거(Trigger) | 가이드 노출 조건을 만족하는 사용자 행동 이벤트 |
| 분류 신뢰도(confidence) | LLM이 user_type을 분류할 때 반환하는 확신 수준 (0.0 ~ 1.0) |
| 임계값(θ) | 분류 신뢰도 채택 기준선. LLM 프롬프트 1차 튜닝 (§9) 결과를 보고 결정 |
| 사용 이력 | 해당 사용자가 누적 생성한 콘텐츠의 분류 이력 (`UserState.classifications`) |
| fallback | 분류 실패 또는 저신뢰도 시 적용되는 대체 동작 |

### 3.3 메모/일정 작성 화면 `ContentCreateScreen` ★ 핵심

**목적**: 사용자가 실제로 메모 또는 일정을 작성하는 화면. 이 화면의 "생성 완료" 버튼 클릭이 LLM 분류 트리거이다.

**구성 요소**

| 요소 | 설명 |
|---|---|
| 탭 전환 | "메모" / "일정" 탭 |
| 제목 입력 필드 | 1줄 텍스트 입력 (필수) |
| 내용 입력 필드 | 멀티라인 텍스트 입력 (선택) |
| 날짜 선택 | 일정 탭에서만 표시 |
| "생성 완료" 버튼 | 하단 고정, 제목이 있을 때만 활성화 |
| "취소" 버튼 | 상단 좌측 |

**입력 검증**
- 제목: 필수, 1자 이상 100자 이하
- 내용: 선택, 최대 2000자
- 날짜: 일정 탭에서 필수

**동작 순서**
1. 사용자가 제목·내용 입력
2. "생성 완료" 버튼 클릭
3. 로컬 상태에 메모/일정 객체 저장
4. **LLM 분류 API 호출** (제목 + 내용을 입력으로 전달)
5. 분류 결과 수신 후 §4.3 Fallback 처리 흐름에 따라 user_type 결정
6. 메인 화면으로 이동

**로딩 처리**
- 분류 API 호출 중 "추천 준비 중..." 표시 (최대 5초)
- 5초 초과 시 분류 없이 메인 화면 이동, user_type은 §4.3 흐름에 따라 결정

**구현 범위**: **A (실구현)**

### 3.4 메인 화면 `MainScreen` ★ 핵심

**목적**: 작성된 콘텐츠를 표시하고, user_type에 따른 맞춤 가이드를 노출한다.

**구성 요소**

| 영역 | 설명 |
|---|---|
| 상단 헤더 | 사용자 이름 + 설정 아이콘 |
| 가이드 카드 영역 | user_type 기반 추천 가이드 (최상단) |
| 콘텐츠 목록 | 생성된 메모·일정 목록 (카드 형태) |
| 하단 FAB | "+ 새로 만들기" 버튼 |

**가이드 카드 구성**
- 아이콘 (user_type별 다름)
- 타이틀 (예: "팀과 함께 작업해보세요")
- 설명 문구 (user_type별 문구, §5.2 참조)
- "시작하기" 버튼 (주요 액션)
- "나중에" 버튼 (카드 닫기)
- "다시 보지 않기" 텍스트 링크

**동작**
- 진입 시 현재 user_type 확인 후 §5.1 매핑 규칙 + §5.4 추천 우선순위에 따라 가이드 로드
- "시작하기" 클릭 시: C 범위 (문서로만 기술, mock 동작)
- "나중에" 클릭 시: 해당 세션에서는 다시 노출 안 함
- "다시 보지 않기" 클릭 시: 로컬 상태에 영구 저장

**구현 범위**: **A (실구현)**

## 4. LLM 분류 로직 명세 ★ 핵심

### 4.1 입력 / 출력

**입력**
```json
{
  "title": "string (1-100자)",
  "content": "string (0-2000자)",
  "type": "memo | schedule"
}
```

**출력**
```json
{
  "user_type": "대학생 | 직장인 | 프리랜서 | 팀 사용자 | unknown",
  "confidence": 0.0 ~ 1.0,
  "reasoning": "string (선택적, 디버깅용)"
}
```

### 4.2 프롬프트 초안

```
당신은 사용자의 생산성 도구 사용 맥락을 분석하는 분류기입니다.

사용자가 작성한 메모 또는 일정의 제목과 내용을 보고, 
다음 4가지 user_type 중 가장 가능성이 높은 하나를 선택하세요.

[분류 기준]
- 대학생: 강의, 시험, 과제, 팀플, 동아리, 학점, 수업 등 학업 맥락
- 직장인: 회의, 보고, 미팅, 업무, 결재, 출장 등 조직 내 업무 맥락
- 프리랜서: 클라이언트, 프로젝트 납품, 견적, 외주, 개인 사업 맥락
- 팀 사용자: "우리 팀", "부서", "공동", "협업" 등 복수 인원 협업 맥락

[판단 불가 기준]
- 맥락이 모호하거나 어느 타입에도 명확히 속하지 않으면 "unknown"으로 반환
- confidence가 임계값 θ 미만이면 반드시 "unknown"으로 반환

[입력]
제목: {title}
내용: {content}
타입: {type}

[출력 형식]
JSON으로만 답변하세요:
{
  "user_type": "대학생|직장인|프리랜서|팀 사용자|unknown",
  "confidence": 0.0-1.0 사이 숫자,
  "reasoning": "판단 근거 한 문장"
}
```

※ 프롬프트 내 임계값 θ는 1차 튜닝 시 구체값으로 치환

### 4.3 Fallback 처리

분류 실패/저신뢰도 시 단계별로 fallback이 작동한다. 모달이나 별도 사용자 입력은 받지 않는다.

#### 분류 흐름

| 순서 | 조건 | 처리 |
|---|---|---|
| 1 | LLM 분류 가능 (confidence ≥ θ) | LLM 결과 사용 |
| 2 | LLM 애매/실패 (confidence < θ, unknown, 타임아웃, 다회 충돌, JSON 파싱 실패) | 사용 이력으로 판단 (§4.6 참조) |
| 3 | 사용 이력도 없음 (신규 유저) | 개인 사용자 fallback |

#### 상세 fallback 매트릭스

| 상황 | 1차 처리 | 2차 처리 (신규 유저) |
|---|---|---|
| API 응답 없음 (5초 타임아웃) | 사용 이력 기반 분류 | 개인 사용자 |
| confidence < θ | 사용 이력 기반 분류 | 개인 사용자 |
| user_type = "unknown" 반환 | 사용 이력 기반 분류 | 개인 사용자 |
| JSON 파싱 실패 | 사용 이력 기반 분류 + 에러 로그 | 개인 사용자 |
| user_type 다회 충돌 | 사용 이력 기반 분류 | 개인 사용자 |
| 이전 분류 결과가 있는 경우 | 신규 분류 confidence가 기존보다 낮으면 기존 유지 | — |

### 4.4 누적 재분류 (선택 구현)

- 첫 3개 생성 데이터까지는 확정 분류 보류
- 3개 누적 시점에 전체를 재분석하여 최종 user_type 확정
- 이 기능은 시간 여유 시 구현, 아니면 **단일 분류만 사용**

### 4.6 사용 이력 기반 판단 규칙

§4.3 Fallback 흐름의 2단계에서 적용된다. 사용 이력 = 해당 사용자가 누적 생성한 콘텐츠의 분류 이력 (`UserState.classifications`).

**판단 규칙**
- 누적 분류 이력 중 confidence ≥ θ로 확정된 결과가 있으면 그 user_type을 사용
- 복수 결과가 있을 경우 최빈값(mode) 사용, 동률이면 최신값
- 확정된 결과가 전혀 없으면 신규 유저로 간주하고 개인 사용자 fallback

**처리 의사코드**
```python
def resolve_user_type_from_history(classifications, theta):
    confirmed = [c for c in classifications if c.confidence >= theta]
    if not confirmed:
        return "개인 사용자"  # 신규 유저
    
    # 최빈값 + 동률 시 최신
    mode_type = mode_with_tiebreak_latest(confirmed)
    return mode_type
```

## 5. 가이드 매핑 엔진

### 5.1 매핑 규칙 (1순위 추천)

| user_type | 1순위 추천 | 액션 버튼 문구 | 대상 기능 |
|---|---|---|---|
| 대학생 | 알림 자동화 | 알림 설정하기 | notification_rule |
| 직장인 | 팀원 초대 | 팀원 초대하기 | team_invite |
| 프리랜서 | 알림 자동화 | 알림 설정하기 | notification_rule |
| 팀 사용자 | 알림 자동화 설정 | 규칙 만들기 | notification_rule |
| 개인 사용자 (기본값) | 협업 메모 공유 | 메모 공유하기 | note_share |

> [!important] 사용 이력 결합 원칙
> 추천은 user_type 단독으로 결정하지 않는다. 1순위 추천 기능을 **이미 사용한 사용자**에게는 해당 가이드를 노출하지 않고, §5.4 우선순위에 따라 다음 미사용 기능으로 이동한다.

### 5.2 가이드 문구 (제안 텍스트)

```yaml
대학생:
  title: "중요한 일정을 놓치지 않게 알림을 설정하세요"
  description: "시험, 과제 마감을 자동으로 리마인드 받을 수 있어요."
  cta: "알림 설정하기"

직장인:
  title: "동료와 함께하면 더 편해요"
  description: "방금 만든 내용을 팀원에게 바로 공유할 수 있어요."
  cta: "팀원 초대하기"

프리랜서:
  title: "마감을 놓치지 않는 알림을 설정해보세요"
  description: "클라이언트 납품 일정을 자동으로 관리할 수 있어요."
  cta: "알림 설정하기"

팀 사용자:
  title: "반복되는 알림을 자동화해 효율을 높이세요"
  description: "주간 회의, 정기 보고 알림을 자동 규칙으로 만들 수 있어요."
  cta: "규칙 만들기"

개인 사용자:
  title: "이 메모를 다른 사람과 공유할 수 있어요"
  description: "링크 하나로 팀원·지인에게 전달할 수 있어요."
  cta: "메모 공유하기"
```

### 5.3 노출 제어 규칙

| 규칙 | 상세 |
|---|---|
| 세션당 최대 1회 | 같은 가이드를 한 세션에서 여러 번 보여주지 않음 |
| "나중에" 거절 시 | 해당 세션에서 다시 표시 안 함 |
| "다시 보지 않기" 시 | 영구 표시 안 함 (로컬 저장) |
| 동시 다중 가이드 금지 | 한 화면에 하나의 가이드만 노출 |
| 가이드 미반응 시 | 30초 후 카드가 반투명 처리 (주의 분산 방지) |

### 5.4 추천 우선순위 (사용 이력 결합)

§5.1의 1순위가 이미 사용된 경우, 다음 우선순위로 이동한다. 개인 사용자 fallback이 가장 일반적인 적용 대상이다.

#### 의사코드

```python
# 개인 사용자 fallback 시 추천 우선순위
if user_type == "개인 사용자":
    if team_invite == 0:
        추천 = "팀 초대"          # 협업 기능 사용률 낮음 (데이터 기반)
    elif notification_rule == 0:
        추천 = "알림 설정"
    else:
        추천 = "메모 공유"
```

#### 설계 원칙

- 아직 팀 기능 안 쓴 사용자 → 팀 초대 추천
- 팀 기능은 썼지만 알림 안 쓴 사용자 → 알림 추천
- **개인화 유지 + 데이터 기반 설계 + 추천 설득력 확보**
